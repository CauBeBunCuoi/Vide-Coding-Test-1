# Pipeline: Init Agent Team

Run this pipeline at the start of any new project to prepare agent files and spawn the team.
Follow every step in order. Do not skip or assume — execute each step explicitly.

## Naming Conventions

- Agent role file name: `<project-kebab>-agent_<role>.md` — e.g. `my-app-agent_frontend.md`
- Derive `<project-kebab>` from the project root folder name in kebab-case

## Hard Rules — Enforce Before Every Run

STOP and do not proceed if any of these violations are found:

1. Any `@import` in the `# Agent Roles` section of `CLAUDE.md` references a `.example.md` file
2. Any real agent file (the one being injected into agents) contains `.example` in its name
3. Any agent role file name does not follow `<project-kebab>-agent_<role>.md` format

Fix all violations before continuing.

---

## Step 1 — Read the Team Roster

Open `CLAUDE.md` and find the `# Agent Roles` section.
Count how many `@import` lines exist in that section. That number is the total agent count.
Note the exact file path each import points to — you will need these in later steps.

---

## Step 2 — Enforce No .example In Imports

Scan every `@import` line in the `# Agent Roles` section of `CLAUDE.md`.
If any import path contains `.example`, STOP — fix it before proceeding.

Real imports must look like:
```
@import .claude/agents/<project-kebab>-agent_frontend.md
@import .claude/agents/<project-kebab>-agent_backend.md
@import .claude/agents/<project-kebab>-agent_tester.md
```

Not like:
```
@import .claude/agents/teammate-a.example.md   ← FORBIDDEN
```

---

## Step 3 — Check If Real Role Files Exist

For each agent import found in Step 1, check the file actually exists:

```bash
ls .claude/agents/
```

If a real role file is missing, copy from the matching `.example.md` template and rename it to follow the `<project-kebab>-agent_<role>.md` convention. Customize the content for this project.

Do NOT modify or delete any `.example.md` file — they are permanent templates.

---

## Step 4 — Update CLAUDE.md Imports If Needed

If any import in `CLAUDE.md` still points to a `.example.md` or an old generic name (e.g. `teammate-a.md`), update it to the correct project-specific name:

```
@import .claude/agents/<project-kebab>-agent_frontend.md
@import .claude/agents/<project-kebab>-agent_backend.md
@import .claude/agents/<project-kebab>-agent_tester.md
```

Also update the `# Agent Team` roster table and delegation examples in `CLAUDE.md` to use the real file names.

---

## Step 5 — Verify Subproject Structure

Each agent has a designated workspace folder. Verify the following exist:

| Agent      | Workspace        | CLAUDE.md                  | Rules folder                    |
|------------|------------------|----------------------------|---------------------------------|
| Teammate A | `taskflow-fe/`   | `taskflow-fe/CLAUDE.md`    | `taskflow-fe/.claude/rules/`    |
| Teammate B | `taskflow-be/`   | `taskflow-be/CLAUDE.md`    | `taskflow-be/.claude/rules/`    |
| Teammate C | `taskflow-test/` | `taskflow-test/CLAUDE.md`  | `taskflow-test/.claude/rules/`  |

### Fill in rule files before spawning

Each subproject's `.claude/rules/` contains placeholder files. Fill them in with project-specific standards before the team starts work:

- `taskflow-fe/.claude/rules/conventions.md` — frontend coding conventions
- `taskflow-fe/.claude/rules/component-standards.md` — component structure and naming
- `taskflow-be/.claude/rules/database.md` — DB schema and migration conventions
- `taskflow-be/.claude/rules/api.md` — REST API design standards
- `taskflow-be/.claude/rules/architecture.md` — backend architecture decisions
- `taskflow-be/.claude/rules/conventions.md` — backend coding conventions
- `taskflow-test/.claude/rules/conventions.md` — test naming, structure, coverage

> **Template note:** The file names above are examples only. The `rules/` folder structure is fixed, but you define the file names to match your project's needs. Add, rename, or remove files as appropriate.

**Rule:** Business rules and coding standards belong in subproject `.claude/rules/` only. The root `.claude/rules/` is for team coordination rules only.

---

## Step 6 — Spawn the Agent Team

With `teammateMode: in-process` set in `settings.json`, teammates run inside your main terminal.
Use `Shift+Down` to cycle between them and `Ctrl+T` to view the shared task list.

Spawn the full team in a single natural language request:

```
Create an agent team with a frontend developer, backend developer, and tester
using the role files in .claude/agents/. Stand by for tasks.
```

Claude Code handles spawning, task list setup, and teammate coordination automatically.
Wait for all teammates to acknowledge their roles before proceeding.

---

## Step 7 — Report To User

Once all teammates are ready, report a summary:

```
Team is ready.

| Agent      | Agent file                          | Role               |
|------------|-------------------------------------|--------------------|
| Teammate A | <project-kebab>-agent_frontend.md   | Frontend Developer |
| Teammate B | <project-kebab>-agent_backend.md    | Backend Developer  |
| Teammate C | <project-kebab>-agent_tester.md     | Tester             |
```

If any teammate fails to acknowledge — re-send their spawn prompt and wait again.

---

## Notes

- `.example.md` files are permanent templates — never inject them into agents, never delete them
- Real agent files must never contain `.example` in their name
- Always update `CLAUDE.md` imports and roster to reflect the real file names before spawning
- Agents use `isolation: worktree` in their frontmatter — Claude Code creates and cleans up worktrees automatically
- `.worktreeinclude` at project root copies gitignored env files into each worktree
- The old `init-worktrees` pipeline is no longer needed
