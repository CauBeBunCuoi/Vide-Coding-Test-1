# Pipeline: Apply Requirements

Run this pipeline after placing project documents in the `.claude/requirements/` folder.
It reads all requirement docs and distributes the relevant content into each subproject's files.

`.claude/requirements/` accepts **any document type, any number of files** — PRDs, specs, ADRs,
API contracts, design docs, meeting notes, or anything else the dev drops in.
The root agent reads everything and uses its own understanding to decide what belongs where.

Follow every step in order. Do not skip or assume — execute each step explicitly.

---

## Hard Rules

- `.claude/requirements/` is READ-ONLY — never write to or modify any file inside it
- Only update subproject files: `CLAUDE.md`, `.claude/rules/`, and `.claude/skills/`
- Do not overwrite existing content — append or merge intelligently
- Do not inject irrelevant content into a subproject that doesn't own it

---

## Step 1 — Detect Active Worktrees

Check whether worktrees have been initialised:

```bash
git worktree list
```

Note which of these paths exist:
- `.worktrees/taskflow-fe`
- `.worktrees/taskflow-be`
- `.worktrees/taskflow-test`

Keep this list — Steps 4, 5, and 6 will write to the main checkout first,
then repeat the same writes to every active worktree.

---

## Step 2 — Scan Requirements Folder

List all files in `.claude/requirements/`:

```bash
ls .claude/requirements/
```

Read every file found. Build a full picture of the project before writing anything.

---

## Step 3 — Extract and Route Per-Subproject Content

Read all documents holistically. Use your own understanding to decide what belongs where.
There is no fixed mapping — extract based on meaning, then route to the correct folder.

### Routing taxonomy

Each subproject has two destination folders:

| Folder | Route content that... |
|---|---|
| `.claude/rules/` | Describes **how to write** the code — behavioral standards, conventions, constraints, patterns. Content that is short, always true, and needed every session. |
| `.claude/skills/<name>/references/` | Describes **what** the system is — screens, flows, API lists, domain models, data schemas, test scope. Large reference content that is only needed when performing a specific task. Place inside the skill directory it supports. |

> **Key distinction:** `rules/` is for behavioral instructions an agent must always follow. `skills/references/` is for large reference content an agent looks up on demand. Never put large domain reference docs in `rules/`.

### Subproject scope guidance (not exhaustive)

| Subproject     | Typically relevant content                                           |
|----------------|----------------------------------------------------------------------|
| `taskflow-fe`  | UI/UX specs, frontend tech stack, component inventory, design system |
| `taskflow-be`  | Backend tech stack, data models, API design, auth, architecture      |
| `taskflow-test`| Test strategy, coverage targets, QA scope, edge cases                |

- If content belongs to more than one subproject, include it in all relevant ones
- If a document is ambiguous, use context clues to infer the correct subproject
- Create new files in the correct folder as needed — do not force content into an existing file if it doesn't fit

## Step 4 — Update Frontend Subproject

**Targets:**
- Main: `taskflow-fe/CLAUDE.md`, `taskflow-fe/.claude/rules/`, `taskflow-fe/.claude/skills/`
- Worktree (if active): same paths under `.worktrees/taskflow-fe/taskflow-fe/`

Route extracted frontend content to the correct folder using the taxonomy from Step 3.
Update `CLAUDE.md` → `## Project Context` with tech stack and folder structure overview.

## Step 7 — Report To User

After all files are updated, report a summary:

```
Requirements applied.

| Subproject     | Locations updated                          | Files                                                   |
|----------------|--------------------------------------------|---------------------------------------------------------|
| taskflow-fe    | main + worktree (if active)                | CLAUDE.md, <your files>                                 |
| taskflow-be    | main + worktree (if active)                | CLAUDE.md, <your files>                                 |
| taskflow-test  | main + worktree (if active)                | CLAUDE.md, <your files>                                 |

Requirements source: .claude/requirements/ (read-only, unchanged)
```

> **Template note:** Replace `<your files>` in the report table above with the actual file names you created or updated. The `skills/` and `rules/` folder structure is fixed, but file names inside are yours to define per project.

If any requirement doc was missing content for a subproject, note it so the user knows what to fill in manually.

---

## Notes

- `.claude/requirements/` accepts any document type, any number of files — no predefined structure required
- The root agent reads all docs and applies content using its own judgment — no rigid file-to-rule mapping
- New rule files can be created if requirements introduce content categories not covered by existing files
- Run this pipeline any time requirements change — it is safe to re-run
- If a rule file already has content, merge new information rather than replacing
- The `.claude/requirements/` folder is the single source of truth — do not duplicate its raw content; extract and translate it into agent-ready files
- If worktrees are active, always write to both main checkout and the worktree — teammates read from their worktree, not from main
- If requirements change after worktrees are active, just re-run this pipeline — no teardown needed
