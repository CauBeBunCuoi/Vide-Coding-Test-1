# Pipeline: Apply Requirements

Run this pipeline after placing project documents in the `requirements/` folder.
It reads all requirement docs and distributes the relevant content into each subproject's context files.

Follow every step in order. Do not skip or assume — execute each step explicitly.

---

## Hard Rules

- `requirements/` is READ-ONLY — never write to or modify any file inside it
- Only update subproject files: `CLAUDE.md` sections and `.claude/rules/` files
- Do not overwrite existing content — append or merge intelligently
- Do not inject irrelevant content into a subproject that doesn't own it

---

## Step 1 — Detect Active Worktrees

Check whether worktrees have been initialised:

```bash
git worktree list
```

Note which of these paths exist:
- `.worktrees/taskflow-fe`
- `.worktrees/taskflow-be`
- `.worktrees/taskflow-test`

Keep this list — Steps 3, 4, and 5 will write to the main checkout first,
then repeat the same writes to every active worktree.

---

## Step 2 — Scan Requirements Folder

List all files in `requirements/`:

```bash
ls requirements/
```

Read every file found. Build a full picture of the project before writing anything.

---

## Step 3 — Extract Per-Subproject Content

From everything you read, identify what belongs to each subproject:

| Subproject     | What to extract                                                      |
|----------------|----------------------------------------------------------------------|
| `taskflow-fe`  | UI/UX specs, frontend tech stack, component inventory, design system |
| `taskflow-be`  | Backend tech stack, data models, API design, auth, architecture      |
| `taskflow-test`| Test strategy, coverage targets, QA scope, edge cases                |

If a piece of information belongs to more than one subproject, include it in all relevant ones.

---

## Step 4 — Update Frontend Subproject

Write to the main checkout first, then repeat for the worktree if it exists.

**Targets:**
- Main: `taskflow-fe/CLAUDE.md` and `taskflow-fe/.claude/rules/`
- Worktree (if active): `.worktrees/taskflow-fe/taskflow-fe/CLAUDE.md` and `.worktrees/taskflow-fe/taskflow-fe/.claude/rules/`

**`CLAUDE.md`** — update `## Project Context`:
- Frontend tech stack and framework
- Folder structure overview
- Key pages or screens

**`.claude/rules/conventions.md`** — fill in:
- Naming conventions, formatting rules, import order

**`.claude/rules/component-standards.md`** — fill in:
- Component structure, prop naming, state management patterns
- Design system tokens, styling approach

---

## Step 5 — Update Backend Subproject

**Targets:**
- Main: `taskflow-be/CLAUDE.md` and `taskflow-be/.claude/rules/`
- Worktree (if active): `.worktrees/taskflow-be/taskflow-be/CLAUDE.md` and `.worktrees/taskflow-be/taskflow-be/.claude/rules/`

**`CLAUDE.md`** — update `## Project Context`:
- Backend tech stack and framework
- Folder structure overview

**`.claude/rules/database.md`** — fill in:
- DB schema conventions, migration standards, query patterns

**`.claude/rules/api.md`** — fill in:
- Endpoint naming, versioning, request/response format, error codes
- Use content from `requirements/api-contracts.md` as the source of truth

**`.claude/rules/architecture.md`** — fill in:
- Layer structure, service boundaries, key architectural decisions

**`.claude/rules/conventions.md`** — fill in:
- Coding conventions, error handling patterns, logging standards

---

## Step 6 — Update Tester Subproject

**Targets:**
- Main: `taskflow-test/CLAUDE.md` and `taskflow-test/.claude/rules/`
- Worktree (if active): `.worktrees/taskflow-test/taskflow-test/CLAUDE.md` and `.worktrees/taskflow-test/taskflow-test/.claude/rules/`

**`CLAUDE.md`** — update `## Project Context`:
- Test framework and tooling
- What is in scope for testing

**`.claude/rules/conventions.md`** — fill in:
- Test naming, file structure, coverage thresholds
- Mock strategy, test data approach

---

## Step 7 — Report To User

After all files are updated, report a summary:

```
Requirements applied.

| Subproject     | Locations updated                          | Files                                                   |
|----------------|--------------------------------------------|---------------------------------------------------------|
| taskflow-fe    | main + worktree (if active)                | CLAUDE.md, conventions.md, component-standards.md       |
| taskflow-be    | main + worktree (if active)                | CLAUDE.md, database.md, api.md, architecture.md, conventions.md |
| taskflow-test  | main + worktree (if active)                | CLAUDE.md, conventions.md                               |

Requirements source: requirements/ (read-only, unchanged)
```

If any requirement doc was missing content for a subproject, note it so the user knows what to fill in manually.

---

## Notes

- Run this pipeline any time requirements change — it is safe to re-run
- If a rule file already has content, merge new information rather than replacing
- The `requirements/` folder is the single source of truth — do not duplicate its raw content; extract and translate it into agent-ready rules
- If worktrees are active, always write to both main checkout and the worktree — teammates read from their worktree, not from main
- If requirements change after worktrees are active, just re-run this pipeline — no teardown needed
